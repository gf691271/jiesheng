# 接声 Android APK 设计规格

日期：2026-07-20

## 目标

交付一个单用途 Android 小工具：用户从系统文件选择器选取 2–3 个音频文件，指定先后顺序，将它们顺序拼接并导出为一个 M4A 文件。应用完全离线，不修改源文件。

项目公开托管在 GitHub，采用 Apache License 2.0，并以 Google 官方开源 Media3 Transformer 作为媒体处理依赖。项目是独立应用仓库，不 fork Media3。

## 成功标准

- Android 8.0（API 26）及以上设备可安装并启动 APK。
- 用户可以一次选择或分次添加 2–3 个音频文件。
- 输入覆盖设备能够解码的 M4A、MP3、WAV、Ogg、FLAC 和 AMR 文件。
- 用户可以用上移、下移和删除操作确定顺序。
- 只有选择 2–3 个有效条目后才允许开始合并。
- 用户通过系统“另存为”界面指定输出位置和文件名。
- 输出为仅含 AAC 音轨的 M4A 文件，播放顺序与界面一致。
- 合并失败时保留全部源文件、清理临时输出并显示可理解的原因。
- 应用不声明网络权限或读取全部存储权限。
- GitHub Release 提供可直接安装的签名 APK，并在项目 `artifacts/` 保留同一构建产物及校验值。

## 首版范围

### 包含

- 系统文件选择器，多选音频。
- 文件名和时长展示。
- 最多三个条目。
- 上移、下移、删除。
- 合并进度和取消。
- M4A 导出。
- 成功、取消和失败状态。
- Android 自适应应用图标。

### 不包含

- 录音、试听、波形、裁剪、音量调整、淡入淡出或交叉淡化。
- 多音轨混音。
- MP3 或其他输出格式。
- 云端上传、账号、历史记录或后台任务队列。
- 超过三个输入文件。

## 产品与视觉

应用名为“接声”。图标采用深色圆角底，两段橙红色声波在中央连接为一条。图标使用 Android 自适应矢量资源，避免位图缩放失真。

界面为单 Activity、单页面原生 Android Views：

1. 顶部显示“接声”和说明“按顺序拼接 2–3 段音频”。
2. 中部显示已选文件列表，每项包含序号、文件名、时长、上移、下移和删除操作。
3. 空状态显示“选择音频”。已有文件时仍可添加，达到三个后禁用添加。
4. 底部主按钮为“合并并导出”；合并期间显示进度并提供取消。
5. 使用系统文件选择器和系统“另存为”界面，不自行仿造文件管理器。

界面不引入 Jetpack Compose；首版使用 AndroidX Activity、ViewModel 和基础 Views，减少依赖和构建体积。

## 架构

- `MainActivity`：渲染状态，发起系统选取和保存操作，不包含媒体算法。
- `MainViewModel`：维护输入顺序、数量限制、合并状态和一次性界面事件。
- `SelectedAudio`：只保存内容 URI、显示名称和时长。
- `AudioMergeEngine`：合并接口，接受有序 URI 列表、临时输出路径和进度回调。
- `Media3AudioMergeEngine`：首版唯一实现，使用 `EditedMediaItemSequence`、`Composition` 和 `Transformer` 顺序处理音频。
- `DocumentMetadataReader`：通过 `ContentResolver` 读取文件名和媒体时长。

`AudioMergeEngine` 是唯一为未来扩展保留的边界。首版不建立插件系统、通用编辑时间线或多模块工程。

## 数据流

1. `OpenMultipleDocuments` 以 `audio/*` 打开系统选择器。
2. 应用验证返回数量、去重，并读取每个 URI 的名称和时长。
3. 用户调整列表顺序。
4. 用户点击“合并并导出”，应用以 `CreateDocument("audio/mp4")` 请求保存位置，默认名为 `接声_yyyyMMdd_HHmm.m4a`。
5. 用户确认位置后，Media3 将有序输入转码并拼接到应用缓存中的临时 `.m4a` 文件。
6. 完成后应用把临时文件复制到系统返回的目标 URI，刷新进度并删除临时文件。
7. 应用显示目标文件名和“完成”。

应用仅持有完成当前操作所需的 URI 权限，不扫描设备文件夹。旋转屏幕时 ViewModel 保留列表和进度；合并期间保持屏幕唤醒。进程被系统终止后不承诺恢复任务。

## 编码策略

- 输入由 Media3/设备的 `MediaExtractor` 与 `MediaCodec` 能力决定。
- 输出固定为 AAC-LC 音频、MP4/M4A 容器。
- 不启用无损直通优化；不同输入格式、采样率或声道数统一走解码和重新编码，以换取可预期的拼接结果。
- 输出不包含视频轨道。

## 错误处理

- 少于两个文件：主按钮禁用并提示还需添加。
- 超过三个文件：不改变原列表，提示最多三个。
- 重复 URI：忽略重复项并提示。
- 文件无法读取、没有音轨或设备不支持解码：指出对应文件并停止操作。
- 保存界面取消：返回可编辑列表，不启动合并。
- 编码、空间不足或目标写入失败：删除临时文件，保留源列表并显示失败信息。
- 用户取消合并：调用 Transformer 取消，清理临时文件并返回列表。

错误信息面向用户描述下一步，不展示堆栈；详细异常只写入本地日志。

## 测试与验证

实现采用测试先行：先运行失败测试，再写最小实现使其通过。

单元测试覆盖：

- 添加、去重和最多三个文件。
- 两个文件时启用合并，一个或四个时禁止。
- 上移、下移、边界不越界和删除。
- 合并状态转换与默认 M4A 文件名。

Android/集成验证覆盖：

- Content URI 元数据读取。
- 用短 M4A、MP3 和 WAV 样本完成不同格式拼接。
- 输出仅含音频轨，总时长接近各输入时长之和，顺序可辨认。
- 导出到系统文档 URI 后文件可重新打开。
- 模拟器上安装、冷启动、选择、排序、合并、取消和导出。

发布前运行完整单元测试、Android lint、release 构建、APK 签名验证和安装验证。任何跳过项必须在发布说明中明确标出。

## 已知边界

Media3 的输入解码能力部分依赖设备厂商提供的编解码器，因此“支持格式”表示应用接受这些常见容器，并在设备可解码时完成转换。首版不捆绑 FFmpeg 软件解码器；遇到设备缺少解码能力时明确失败，而不是扩大 APK。
